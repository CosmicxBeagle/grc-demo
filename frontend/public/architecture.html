What is GRC — and What Does This Platform Do?

Plain-English context for non-technical readers before we dive into the technical details.

🏦
GRC stands for Governance, Risk & Compliance
Companies — especially those in finance, healthcare, and publicly traded industries — are required by law and regulation to prove that they have controls in place to prevent fraud, data breaches, and operational failures. A control is simply a rule or process the company follows, like "two people must approve every wire transfer over $10,000" or "access logs are reviewed every week."
✅
This platform is where those controls get tested
Once a year (or more often), auditors and internal testers must prove that every control is actually working — not just written down. They gather evidence, run test procedures, and sign off on their findings. If a control isn't working, that's called a deficiency, and it must be fixed and tracked until it's resolved.
⚠️
Why it matters
Failing an audit can result in regulatory fines, loss of operating licences, and reputational damage. This platform exists to make the testing process systematic, traceable, and auditor-ready — replacing spreadsheets and email chains with a structured, role-based workflow with a full audit trail.
👥

Who Uses This Platform

Five user roles — each with different levels of access. The system enforces these boundaries automatically.

Role Plain-English Description Can Test Can Review Can Admin Can View Risks
Admin Full control over everything. Sets up users, policies, and the overall programme. ✔✔✔✔
GRC Manager Runs the compliance programme. Creates test cycles, assigns work, reviews results. ✔✔–✔
GRC Analyst Does the day-to-day testing work — gathers evidence, fills in workpapers, submits for review. ✔––Read
Auditor External or internal auditor. Can read everything but cannot change records — read-only access. –Read–Read
Control Owner The business person responsible for a specific control. Can view and respond to their own items. Own–––
How the system enforces this: When a user logs in they receive a JWT (a secure digital token) that contains their role. Every API request sends this token, and the backend checks permissions before doing anything. The frontend also hides menu items and buttons that a role isn't allowed to use.

What Alembic migrations are
Think of migrations like a version-controlled change log for the database structure. When a developer adds a new field (like "root cause" on a deficiency), they write a migration script. That script is run once on the production database to add the field — without deleting any existing data. Migrations 001–008 represent every structural change made to date.

All Frontend Pages

The frontend is built with Next.js 14 using the App Router — a React framework. Every page is a React component. Data is fetched from the backend via a central api.ts file using Axios, with the user's token automatically attached to every request.

Auth flow — how a user gets in
flowchart LR A["🌍 User opens\nthe app"] --> B B["/login page\nEnter username\n& password"] -->|"POST /auth/token"| C C["FastAPI checks\ncredentials\nagainst database"] -->|"Returns JWT token\n+ user profile"| D D["Token saved to\nsessionStorage\n(cleared on tab close)"] --> E E["App Shell loads\nSidebar + page content\nbased on user role"] E -->|"Every API call includes\nAuthorization: Bearer token"| F["Backend API\nverifies identity\non every request"]
sessionStorage vs localStorage: The token is stored in sessionStorage, which means it is automatically cleared when the browser tab is closed. This is more secure than localStorage (which persists until manually cleared) because it limits the window in which a stolen token could be replayed.

All pages below sit inside the App Shell — a wrapper that checks the user is logged in, then renders the sidebar navigation and the page content side by side. The sidebar adapts to the user's role (Admin-only items are hidden from other roles).

Main Navigation

🗂 My Work new /my-work

A personal dashboard that pulls together everything that needs this user's attention right now — across all test cycles and all deficiencies. Think of it as an inbox for compliance work.

  • My Testing Assignments — controls this user has been assigned to test
  • Review Queue — test results waiting for this user to approve or reject
  • My Deficiencies — open issues where this user is the remediation owner
  • Pending Approvals — items in the approval workflow awaiting a decision
  • Shows "You're all caught up!" when there is nothing outstanding
📊 Dashboard /dashboard

A high-level overview of the programme's health at a glance — for managers and executives who need a summary without diving into individual records.

  • KPI cards — total controls, open deficiencies, test cycle completion rate
  • Status distribution charts — how many tests are in each state
  • Recent activity feed — latest events across the platform
🛡 Control Library /controls

The master list of every internal control the company has documented. A control is a rule or process designed to prevent a specific risk. All testing flows from this library.

  • Browse, search, and filter all controls
  • Create and edit controls with description, owner, and frequency
  • Each control links to its test cycles and deficiency history
  • Controls are the foundation — nothing gets tested that isn't in this library
🔖 SOX Scoping /sox

SOX (Sarbanes-Oxley) is a US law that requires public companies to test specific financial reporting controls every year. This page is where managers mark which controls are "in scope" for SOX and therefore must be tested.

  • Mark controls as SOX in-scope or out-of-scope
  • Assign control owners responsible for each SOX control
  • Set review periods and frequency (quarterly, annual)
  • Drives which controls must appear in test cycles each year
🔬 Test Cycles updated /test-cycles

A Test Cycle is a formal testing period — for example, "Q1 2026 SOX Testing." Within each cycle, individual controls are assigned to testers. This page lists all cycles; clicking one opens its detail page.

  • Create cycles with a name, date range, and status
  • Assign controls to specific testers and reviewers
  • Track overall cycle progress at a glance
  • Cycle detail page shows all assignments with their current status
📎 Evidence /evidence

Auditors don't just take your word for it — they need proof. This page manages all supporting documents uploaded to support test results: screenshots, reports, exports, and signed approvals.

  • Upload files linked to a specific control or assignment
  • Track who uploaded what and when (full chain of custody)
  • Auditors can download all evidence for their review
  • Allowed file types and size limits are enforced on upload
⚠️ Deficiencies updated /deficiencies

When a control is found to not be working as intended, a deficiency is logged. This page is the master register of all deficiencies — open, in remediation, and closed. Click any row to expand it and manage the fix plan.

  • Table with clickable rows — expand to see full detail
  • Remediation Details (new): root cause, business impact, owner, evidence of closure
  • Milestones (new): break the fix into tasks with due dates, assignees, and status
  • Milestone count badge visible on collapsed row
  • Status lifecycle: open → in remediation → remediated → risk accepted
🚫 Exceptions /exceptions

Sometimes a company knowingly cannot comply with a control for a period of time — maybe a system is being replaced. An exception is a formal, time-limited acknowledgement of that gap. It's not a deficiency (that's an unintended failure) — it's a documented risk acceptance.

  • Log exceptions with a reason, risk rating, and expiry date
  • Exceptions must be approved through the approval workflow
  • Expired exceptions are flagged automatically
  • Auditors can see all exceptions and their approvals
✔ My Approvals /approvals

Certain actions in the system — like closing a deficiency or approving an exception — require a second person to sign off. This page shows everything currently waiting for this user's approval decision.

  • Lists all pending approval requests assigned to this user
  • Approve or reject with an optional comment
  • Decision is timestamped and logged in the audit trail
  • Approval policies (configured by admin) define who must approve what

Test Cycle Detail Page — deep dive
The Cycle Detail page (/test-cycles/[id]) is the heart of the testing workflow. Each control assignment is shown as a collapsible row. Click to expand and access five integrated panels:
📄Workpaper Section updated

The tester's formal documentation of what they did and what they found. Equivalent to audit workpapers in a paper-based world.

  • Testing Steps — what procedure was followed
  • Sample Details — which transactions or records were tested
  • Walkthrough Notes — observations during a walkthrough with the control owner
  • Conclusion — the tester's overall finding
  • Evidence Request — what documents have been requested and by when
  • Collapsible using the browser's native <details> element — shows a "filled" badge when any field has content
☑Checklist Panel new

A step-by-step checklist of tasks that must be completed for this assignment. Ensures nothing is missed during testing.

  • Progress bar shows X of Y steps completed
  • Toggle individual items complete/incomplete
  • Add custom checklist items for this assignment
  • Delete items that aren't applicable
  • Checklist item count badge shows on the collapsed row header
📤Submit for Review updated

When the tester is done, they formally submit their work for a second-person review. This replaces ad-hoc email chains with a tracked handoff.

  • Tester adds a sign-off note summarising their work
  • Status changes from "in progress" to "needs review"
  • Records who submitted and the exact timestamp
  • Submission receipt banner appears in the assignment row
  • Reviewer is notified that work is waiting
✅Reviewer Decision updated

The reviewer examines the workpaper and evidence, then makes one of three decisions. This creates a formal, signed-off record.

  • Approve — work is complete and satisfactory → status becomes "complete"
  • Return — more work is needed → status returns to "in progress" so tester can fix it
  • Fail — control has failed testing → status becomes "failed," deficiency should be raised
  • Decision, note, reviewer name, and timestamp all recorded
  • Colour-coded receipt banner shows the outcome
⚠Deficiencies Panel

Issues discovered during testing are logged directly from the assignment — keeping the deficiency linked to the specific test and control where it was found.

  • Lists all deficiencies already linked to this assignment
  • Create a new deficiency without leaving the assignment page
  • Link is maintained: deficiency knows which assignment and control it came from

Admin Section (visible to Admin role only)

📋 Audit Trail admin /audit-logs

Every significant action taken in the system is permanently logged here — who did what, to which record, and exactly when. This is required by SOX and most compliance frameworks. It cannot be edited or deleted.

  • Structured logs with user, action, entity type, entity ID, and timestamp
  • Filter by user, date range, or action type
  • SIEM-ready format (can be exported to security monitoring tools)
  • Read-only — no one can alter the audit trail, not even admins
👤 Users admin /settings/users

Create and manage user accounts. Assign roles. Deactivate accounts when people leave. This is the only place where role assignments are made.

  • Create users with username, email, display name, and role
  • Change a user's role (takes effect immediately on next login)
  • View last login times
  • Deactivate accounts (does not delete history — preserves the audit trail)
⚙ Approval Policies admin /settings/approvals

Configure which actions require approval and who is authorised to approve them. For example: "Closing a deficiency requires a GRC Manager to approve."

  • Define approval rules per entity type (deficiency, exception, risk, etc.)
  • Set required approver role or specific users
  • Single or multi-stage approval chains
  • Changes take effect immediately for new approval requests

Risk Management Section

🏗 Assets /assets

An asset is anything valuable that could be harmed — a system, a process, a data set, or a team. This is the starting point for risk management: you can only assess risk to something once you've identified what matters.

  • Register all business assets with type, owner, and criticality rating
  • Assets are linked to threats (what could harm them) and risks (the assessed likelihood & impact)
⚡ Threats /threats

A threat is something that could cause harm to an asset — a cyberattack, a natural disaster, a rogue employee, a vendor failure. Threats are assessed independently of their likelihood, which is captured in the Risk Register.

  • Catalogue of all identified threats, categorised by type
  • Each threat is linked to the assets it could affect
  • Threat entries feed into risk assessments
⚖ Risk Register /risks

The Risk Register is a living record of every assessed risk — combining a threat with an asset to produce a risk with likelihood, impact, and an overall score. Risks can be parent/child (a group with sub-risks) and can be filtered and sorted by age, score, and owner.

  • Hierarchical table — parent risks with expandable child risks
  • Inherent risk score (before controls) vs residual score (after controls)
  • Age column with colour-coded aging buckets (<30d, 30–90d, 90–180d, 180d+)
  • Treatment plans attached to each risk
  • Export to spreadsheet
  • Expandable detail row shows full description, scores, ownership, and linked controls
📅 Risk Reviews /risk-reviews

Risks can't just be assessed once and forgotten. Risk Reviews are periodic check-ins where the risk owner confirms whether the risk has changed, been mitigated further, or has worsened.

  • Scheduled review workflow with due dates
  • Reviewer confirms or updates the risk's current status and score
  • Review history logged with timestamps
  • Overdue reviews are flagged on the dashboard
